# Core Project

## Overview

The **core** project contains generic, reusable utilities and systems for building applications with EnTT (Entity Component System). This is a foundational library that provides essential infrastructure for window management, event handling, application layers, serialization, runtime reflection, and undo/redo functionality.

**Important**: Everything in the core project is designed to work generically for any application that runs on EnTT. **It must remain generic and application-agnostic.** App-specific code and classes belong in the `app` folder, not in `core`.

## Key Features

### 1. Window Management
- **Dependencies**: GLFW
- **Location**: `Window.h`, `Window.cpp`
- **Description**: Provides cross-platform window creation and management using GLFW
- **Key Classes**: 
  - `Core::Window` - Window wrapper with event callback support

### 2. Event System (EventBus)
- **Location**: `event/EventBus.h`
- **Description**: Type-safe, callback-based event system for decoupled component communication
- **Key Classes**:
  - `Core::EventBus` - Central event dispatcher
  - `Core::IEvent` - Base interface for all events
- **Usage**:// Register a callback
REGISTER_CALLBACK(eventBus, MouseDownEvent, OnMouseDownEvent);

// Push an event
eventBus.PushEvent(MouseDownEvent{button, x, y});

// Handle all queued events
eventBus.HandleEvents();- **Event Types**:
  - Mouse events: `MouseDownEvent`, `MouseUpEvent`, `MouseMoveEvent`, `MouseScrollEvent`
  - Keyboard events: `KeyDownEvent`, `KeyUpEvent`, `KeyCharacterEvent`
  - Window events: `WindowResizeEvent`, `WindowCloseEvent`
  - Application events: `ApplicationCloseEvent`

### 3. Application Layer System
- **Location**: `IApplicationLayer.h`, `Application.h`
- **Description**: Layer-based architecture for organizing application logic
- **Key Classes**:
  - `Core::IApplicationLayer` - Interface for application layers
  - `Core::Application` - Main application class with layer stack
  - `Core::LayerContext` - Service locator for sharing dependencies between layers
- **Usage**:Application app(specs);
app.PushLayer<MyLayer>();
app.Run();
### 4. Logger
- **Location**: `Logger.h`
- **Description**: Thread-safe, stream-based logging system with console, file, and GUI output support
- **Key Features**:
  - Stream-based API with zero-overhead when disabled
  - Multiple log levels (Trace, Debug, Info, Warn, Error, Critical)
  - GUI integration with configurable history for ImGui display
  - Automatic capture of file, line, function, and timestamp
- **Usage**:LOG_TRACE() << "Message: " << value;
LOG_ERROR() << "Error: " << exception.what();
Core::Log::EnableGuiLogging(true);
auto logs = Core::Log::GetGuiLogs();
### 5. UUID System
- **Location**: `UUID.h`
- **Description**: Stable, time-ordered unique identifier system for entity tracking
- **Key Features**:
  - 64-bit UUIDs ordered by creation time
  - Persists across entity recreation (unlike `entt::entity` IDs)
  - **Required for undo/redo system** - provides stable entity identity
  - Serialization and hash support for use in containers
- **Usage**:auto entity = registry.create();
registry.emplace<Core::UUID>(entity); // Required for undo
### 6. Serialization
- **Dependencies**: Cereal
- **Location**: `serializer/Serializer.h`
- **Description**: Archive-based serialization system using Cereal library
- **Key Features**:
  - Binary and XML archive support
  - Custom serialization for EnTT components
  - Scene and entity serialization

### 7. Runtime Reflection
- **Location**: `reflection/Reflection.h`, `reflection/ReflectionImpl.h`, `reflection/reflectionPathParser.h`
- **Description**: Runtime type information and property access system built on EnTT meta
- **Key Features**:
  - Component and field introspection
  - Path-based property access (e.g., `"Transform.Position.x"`)
  - Compile-time and runtime path parsing
  - Integration with EnTT's meta system
- **Usage**:// Compile-time path parsing
constexpr auto path = PATH_FULL_CONSTEXPR("Transform.Position.x");

// Runtime path parsing
auto path = reflection::ParseFullPathRuntime("Transform.Position.x");
### 8. Undo/Redo Management
- **Location**: `applicator/` folder
- **Description**: Comprehensive undo/redo system with field operators and entity snapshots
- **Key Components**:
  - `UndoManager<FieldTypes>` - Template-based undo/redo stack manager
  - `Applicator<FieldTypes, ComponentTypes>` - High-level API for undoable operations
  - `SelectionArchive.h` - Entity snapshot system for structural operations
  - `Clipboard.h` - Copy/paste support with undo integration
  - Operation types in `ops/`:
    - `SetFieldOp` - Undo/redo for field changes
    - `StructuralOps.h` - Create/delete entity operations
    - `IOp.h` - Base operation interface

**?? CRITICAL: The entire undo folder is a HEADER-ONLY LIBRARY**
- **Cannot be compiled as a separate project or static library**
- **Must be included as headers** in projects that use it
- **Requires template specification** of the types used by the undo manager
- Each application must define its own `FieldTypes` and `ComponentTypes` variant/tuple
- **Requires Core::UUID component** on all entities for stable identity tracking

**Usage Example**:// Define application-specific types (in app, not core!)
using AppFieldTypes = std::variant<int, float, Vec2, /* ... */>;
using AppComponentTypes = std::tuple<Transform, Mesh, /* ... */>;

// Create undo manager and applicator
Core::UndoManager<AppFieldTypes> undoManager;
Core::Applicator<AppFieldTypes, AppComponentTypes> applicator(undoManager);

// Set a field with undo support
applicator.SetField(entity, "Transform.Position.x", 10.0f);

// Undo/Redo
undoManager.Undo();
undoManager.Redo();
### 9. Rendering Utilities
- **Location**: `renderer/` folder
- **Description**: Command-based rendering abstraction
- **Key Classes**:
  - `DrawCommandRecorder` - Records drawing commands
  - `DrawCommandBuffer` - Stores rendering commands
  - `IRenderer` - Renderer interface
  - `Renderer_ImGui` - ImGui-based renderer implementation

### 10. Other Utilities
- **BrowserWindow**: `BrowserWindow.h` - Native file dialog support for opening/saving files

## Design Principles

1. **Generic and Reusable**: All code in core must be application-agnostic
2. **Template-Based**: Heavy use of templates for type safety and flexibility
3. **Header-Only Where Needed**: The undo system is header-only to support template specialization
4. **EnTT Integration**: Designed to work seamlessly with EnTT's ECS and meta systems
5. **Event-Driven**: Decoupled architecture using the EventBus
6. **Layer-Based**: Modular design using application layers

## Important Notes

### What Belongs in Core
? Generic utilities and systems  
? Window management  
? Event system infrastructure  
? Application layer framework  
? Serialization utilities  
? Reflection system  
? Undo/redo framework  
? Renderer abstractions  
? Logging system  
? UUID generation and tracking  

### What Does NOT Belong in Core
? Application-specific logic  
? Scene management (belongs in `app`)  
? Specific component types (belongs in `app`)  
? Game/editor-specific features  
? UI panels (belongs in `app`)  

### Undo System Special Considerations
The undo system (`applicator/` folder) is **header-only** because:
- It requires template specialization with application-specific types
- Each application defines its own `FieldTypes` variant
- Cannot be pre-compiled as a static library
- Must be included as headers in consuming projects

**?? CRITICAL: Undo Manager REQUIRES Core::UUID Component**
- All entities that participate in undo/redo **must** have a `Core::UUID` component
- The UUID provides stable identity when entities are destroyed and recreated
- Without UUIDs, the undo system cannot properly track and restore entities

## Usage Guidelines

When working with the core project:

1. **Keep it Generic**: If you're adding application-specific code, it belongs in `app`, not `core`
2. **Use Templates**: Prefer templates over runtime polymorphism for type safety
3. **Follow EnTT Patterns**: Leverage EnTT's meta system for reflection
4. **Document Public APIs**: All public interfaces should be well-documented
5. **Header-Only Undo**: Remember that undo system files are headers, not implementation files

## Integration Example
// In your application (app folder, not core!)
#include "core/Application.h"
#include "core/applicator/Applicator.h"
#include "core/event/EventBus.h"

// Define app-specific types
using AppFieldTypes = std::variant<int, float, std::string, Vec2>;
using AppComponentTypes = std::tuple<Transform, Renderer, Physics>;

// Create application
Core::ApplicationSpecs specs;
specs.windowTitle = "My Application";
Core::Application app(specs);

// Register services
auto& ctx = app.GetContext();
ctx.Register<SceneManager>(std::make_shared<SceneManager>());

// Add layers
app.PushLayer<GameLayer>();
app.PushLayer<UILayer>();

// Run
app.Run();
